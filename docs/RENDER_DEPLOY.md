# WorkCrew backend deployment on Render

This is the exact, current setup. The backend is the Fastify service in
apps/api. The repository is a monorepo, so the build compiles the shared
contracts first, then the API. Render reads render.yaml (a Blueprint) at the
repo root, so you do not type the build or start commands by hand.

## Quick facts

1. Deployment method: Blueprint (render.yaml at the repo root). Not a manual Web
   Service, not Docker.
2. Runtime: Node. No Python and no Docker are needed for the server. The Windows
   helper runs on the user's PC, never on Render.
3. Stripe webhook path: `/v1/billing/webhook`
4. Final webhook URL after deploy: `https://YOUR-RENDER-URL.onrender.com/v1/billing/webhook`
5. Health check path: `/health` (Render sets this from the blueprint).
6. Build command (set by the blueprint, do not type it):
   `npm install && npm run build -w @workcrew/contracts && npm run build -w @workcrew/api`
7. Start command (set by the blueprint, do not type it):
   `node apps/api/dist/server.js`
8. Root directory: leave blank (the repo root).
9. Production database: Supabase Postgres, via DATABASE_URL. The local app uses a
   SQLite file, but Render wipes local disk on every deploy, so production must
   use Postgres. The backend creates its tables automatically on first boot.

## Environment variables

Set automatically by the blueprint (you do nothing for these):
- NODE_ENV = production
- HOST = 0.0.0.0
- AUTH_MODE = local
- BILLING_MODE = stripe
- WORKCREW_ALLOWED_ORIGINS = app://workcrew
- WORKCREW_LOCAL_AUTH_SECRET = generated by Render

You paste these in the Render dashboard (each is marked "sync: false" in the
blueprint, so Render will prompt you for them):
- DATABASE_URL = your Supabase Postgres connection URI (it contains the database
  password). Backend only.
- ANTHROPIC_API_KEY = your AI provider key (starts with sk-ant-).
- SUPABASE_SERVICE_ROLE_KEY = your Supabase service role key (secret).
- STRIPE_SECRET_KEY = your Stripe secret key (sk_test_ for testing, sk_live_ for
  real charges).
- STRIPE_WEBHOOK_SECRET = added AFTER the first deploy, then redeploy (see below).
- STRIPE_PRO_MONTHLY_PRICE_ID
- STRIPE_PRO_YEARLY_PRICE_ID
- STRIPE_ULTRA_MONTHLY_PRICE_ID
- STRIPE_ULTRA_YEARLY_PRICE_ID
- SUPABASE_URL = your Supabase project URL (public, safe to paste).
- SUPABASE_ANON_KEY = your Supabase anon key (public, safe to paste).

## Order of operations

1. Push the repository to GitHub (private).
2. In Render: New, then Blueprint, then select the repository. Render reads
   render.yaml and shows the service plus the list of values to paste.
3. Paste every value above except STRIPE_WEBHOOK_SECRET.
4. Deploy. When it goes live, copy the service URL
   (https://something.onrender.com).
5. In Stripe (test mode first), add a webhook endpoint at that URL plus
   /v1/billing/webhook. Select these events: checkout.session.completed, and the
   customer.subscription created/updated/deleted/paused/resumed events.
6. Copy the webhook signing secret (starts with whsec_), paste it into Render as
   STRIPE_WEBHOOK_SECRET, and trigger a redeploy.
7. In the desktop app, open Settings and set the Backend address to your Render
   URL.

## Notes

- Until STRIPE_WEBHOOK_SECRET is set, the server still starts; it just rejects
  webhook events until the secret is in place, which is why step 6 exists.
- Supabase gives you both the project URL and keys (Project Settings, API) and
  the Postgres connection string (Project Settings, Database). The service role
  key and the Postgres URI are secret; the URL and anon key are public.
